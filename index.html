데이터베이스 안 용어정리

데이터베이스에서 사용되는 다양한 용어정리

테이블 (Table)

행과 열로 이루어진 데이터의 집합을 테이블이라고 한다.

원자성 (Atomicity), 일관성 (Consistency), 격리성 (Isolation), 내구성 (Durability)

행 (Row)

테이블에서 데이터들을 가로로 묶은 값

행은 한 객체에 대한 정보를 가지고 있고, 레코드(Record) 또는 튜플(Tuple) 라고 불린다.

열 (Column)

테이블에서 데이터들을 세로로 묶은 값

테이블의 속성을 의미하며 같은 도메인(Domain) 으로 이루어져 있다.
속성(Attribute) 라고도 불린다.

도메인 (Domain)

도메인 = 속성 값의 집합

해당 속성 값들을 묶어 사전에 정의해 놓으면, 새로운 속성을 정의할 때,
데이터 타입을 새로 명시 하지 않아도 됨

ex.) 나이 : int(5) , 이름 : varchar(20)

스키마 (Schema)

데이터베이스의 구조를 전반적으로 기술한 것

외부 스키마

사용자의 입장에서 정의한 데이터베이스의 논리적 구조.
데이터들을 어떤 형식, 구조, 화면을 통해 사용자에게 보여줄 것인가에 대한 명세를 말하며
하나의 데이터베이스에는 여러개의 외부 스키마가 있을 수 있다.
일반 사용자에게는 질의어를 이용해 DB를 쉽게 사용할 수 있도록 하고
응용 프로그래머는 언어를 사용해서 DB에 접근하도록 한다.

개념 스키마

조직체 전체를 관장하는 입장에서 DB를 정의한 스키마.
DB에 대한 모든 논리적 구조를 기술하기 때문에 데이터베이스에 하나만 존재하며,
통상 스키마 라고 하면 개념 스키마를 일컫는다.

내부 스키마

데이터베이스가 어떻게 저장장치에 저장될 지에 대한 명세.
물리적인 저장장치와 데이터베이스 간의 관계를 정의하므로
시스템 프로그래머나 시스템 설계자가 보는 관점의 스키마이다.

RDB(Relational Database)에서 M:N 관계 관리 (Spring)

M:N 관계 해소 및 활용

M:N 관계란?

M:N 두 개의 테이블이 서로의 행에 대해서 여러개로 연관 되어 있는 상태를
다대다(M:N)관계라고 합니다.

M:N 관계가 생기는 상황

ex.) 개발자가 여러개의 프로젝트를 관리할 때

개발자는 여러 개의 프로젝트를 관리하고, 프로젝트에는 한 명의 개발자가 아닌 여러명의 개발자가 진행할 때
M:N 관계가 발생


※ 1:N의 관계가 아닌 M:N 관계가 발생함.

M:N 관계 해소하기

Relation Table을 따로 생성하여 M:N관계를 해소

project A, B, C와 개발자 D1, D2, D3가 있을 때


Relation Table 생성

M:N 관계 테이블 활용

INSERT, SELECT, DELETE

M:N 관계 테이블에서 생성하기 (INSERT)

ex.) 개발자 D1이 A 프로젝트를 관리하게 되는 경우

result : 0 번째 값인 D1 | A 생성

M:N 관계 테이블에서 조회하기 (SELECT)

ex.) 프로젝트 A를 관리하고 있는 개발자를 조회하는 경우

result : D1, D2 조회

M:N관계 테이블에서 삭제하기 (DELETE)

ex.) 프로젝트 B를 관리하던 D3가 관리를 종료할 경우

REST-API URL 설계를 위한 규칙

REST-API

REST API란?

REST(Representational State Transfer)의 약자로
자원을 이름으로 구분하여 해당 자원의 상태를 주고받는 모든 것을 의미

REST API의 작동 방식

REST API는 HTTP 요청을 통해 통신함으로써 리소스 내에서
레코드의 쓰기, 읽기, 수정 삭제 등의 기능을 수행

REST-API URL 규칙

1. URL 마지막에는 슬래시를 포함하지 않는다.

후행 슬래시 (/)는 의미 없지만 혼동을 일으킬 수 있음.

ex.)

http://api.canvas.com/shapes (o)

http://api.canvas.com/shapes/ (x)

2. 계층적 관계를 표시할 때는 구분 기호로 (/)를 사용

슬래시는 리소스 간의 계층적 관계를 나타내기 위해 URL 경로에 사용합니다.

ex.)

http://api.canvas.com/shapes/polygons/squares

3. 밑줄 (_)은 사용 X, 가독성을 높이기 위한 하이픈(-)을 사용

공백이나 하이픈이 사용되는 URL 모든 부분에 사용
여러가지 이유로 밑줄이 가려지거나 숨겨질 수 있음

ex.)

http://api.example.com/blogs/guy-levin/posts/this-is-my-first-post

4. URL 경로에서는 소문자를 사용

대문자 사용으로 인해 문제가 발생할 수 있으므로 소문자를 사용
※ RFC 3986은 체계 및 호스트 구성 요소를 제외하고 URI를 대소문자를 구분하는 것으로 정의

5. 파일 확장자는 URL에 포함 X

대신 Content-Type 헤더를 통해 전달되는
미디어 유형에 의존하여 본문의 콘텐츠를 처리하는 방법을 결정

ex.)

http://api.college.com/students/3248234/courses/2005/fall.json (X)

http://api.college.com/students/3248234/courses/2005/fall (O)
